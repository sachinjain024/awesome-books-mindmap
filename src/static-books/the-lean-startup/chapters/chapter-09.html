Chapter 9: Batch

Part Three: Accelerate

This chapter explores why small batches beat large batches in startup environments. Drawing from lean manufacturing principles, Ries shows how working in small batches accelerates learning, reduces waste, and helps startups move through the Build-Measure-Learn loop faster.

The Envelope Stuffing Experiment

Ries opens with a simple thought experiment: imagine you need to stuff 100 envelopes with letters. Which approach is faster?

Two Approaches to Envelope Stuffing

Large batch: Fold all 100 letters, then stuff all 100 envelopes, then seal all 100, then stamp all 100.

Small batch: Complete one envelope at a time—fold, stuff, seal, stamp—before moving to the next.

Intuition says large batches should be faster (less "switching costs"), but experiments show small batches consistently win. Why? Because large batches hide problems. If you discover the letters don't fit the envelopes, you've already folded all 100.

Key Insight

Small batches enable faster feedback. You discover problems immediately instead of after investing significant work in the wrong direction.

Small Batches in Product Development

The same principle applies to software and product development. Large batches—big releases with many features—seem efficient but create problems:

Problems with Large Batches

  • Delayed feedback: You don't learn what works until everything ships
  • Hard to diagnose: When something fails, you don't know which feature caused it
  • Inventory buildup: Unshipped features are "inventory" that might never be used
  • Integration nightmares: Big merges create conflicts and bugs

Case Study: Toyota's Andon Cord

At Toyota, any worker can pull the "andon cord" to stop the production line when they spot a problem. This seems inefficient—stopping the whole line?—but it prevents defects from piling up. Problems are fixed immediately, not at the end.

The startup equivalent: ship small changes, get immediate feedback, fix problems before they compound.

Continuous Deployment

The ultimate expression of small batch thinking is continuous deployment: shipping code to production as soon as it's ready, sometimes dozens of times per day.

Continuous Deployment Benefits

  • Immediate feedback on every change
  • Easier to diagnose problems (you know exactly what changed)
  • Lower risk per deployment (small changes are easier to roll back)
  • Faster cycle through Build-Measure-Learn loop
"The ability to learn faster from customers is the essential competitive advantage that startups must possess."
— Eric Ries

The IMVU Story: Continuous Deployment

IMVU was one of the pioneers of continuous deployment, and Ries shares their journey.

Case Study: IMVU's Deployment Pipeline

IMVU deployed code to production 50 times per day. Key practices:

  • Automated testing: Every change passed through automated tests before deployment
  • Immune system: Automated monitoring that could detect and roll back bad deployments
  • Feature flags: New features could be enabled for small groups first
  • Real-time metrics: Immediate visibility into how changes affected key metrics

Pull, Don't Push

Another lean manufacturing concept: pull systems vs. push systems.

Push vs. Pull

Push: Work is pushed through the system according to a schedule, regardless of whether downstream processes are ready.

Pull: Work is pulled by the next stage when it's ready, ensuring nothing piles up.

In startups, a pull approach means building features only when you have evidence customers want them, not because they're on a roadmap. Customer demand "pulls" development.

Reducing Batch Size

If you're currently working in large batches, how do you transition to smaller ones?

Steps to Reduce Batch Size

  1. Map your current process: How long does it take to go from idea to customer hands?
  2. Identify the biggest delays: Where does work pile up waiting?
  3. Automate repetitive work: Testing, deployment, monitoring
  4. Start smaller: Can you ship half as much, twice as often?
  5. Measure cycle time: Track how long each batch takes from start to finish

Key Takeaways

  • Small batches beat large batches because they enable faster feedback
  • Large batches hide problems until it's expensive to fix them
  • Continuous deployment is the ultimate form of small batch thinking
  • Pull systems (build what customers want) beat push systems (build according to plan)
  • The faster you can complete the Build-Measure-Learn loop, the faster you learn
